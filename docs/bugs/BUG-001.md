# BUG-001 — Invite creation response exposes invite token

- Severity: Medium
- Area: Auth / Users & Access / Invites
- Status: Fixed
- MVP UAT classification: Known issue, not a blocker
- Fixed in: TASK-047

## Classification review

Phase 8 follow-up reviewed BUG-001 classification only. No code, env, schema, or source files were modified. No secret values were printed.

Findings:

1. The invite token is exposed in the immediate `POST /api/auth/invites` create-invite response. No later API route was found that retrieves invite tokens again.
2. The secret is exposed as a top-level `token` field in that one-time create response to an authenticated admin-only endpoint.
3. The database schema stores `UserInvite.tokenHash` only; no plaintext invite token column exists.
4. AuditLog records for the Phase 8 run did not contain token-like secret entries in `beforeData`, `afterData`, or `metadata`. Frontend-visible logs do not select AuditLog sensitive data, and no frontend console/local/session storage logging of the invite token was found in the reviewed source.
5. Severity is **Medium** because the token is only exposed once to admin during invite creation, but this violates the stated Phase 8 security expectation that invite token values are not returned after creation.
6. This is a **KNOWN ISSUE**, not a **BLOCKER**, for MVP UAT unless the acceptance bar is interpreted as forbidding even one-time admin delivery of invite tokens.

## Steps to reproduce

1. Authenticate as an admin over the production HTTPS API.
2. Request CSRF token.
3. `POST /api/auth/invites` with a new isolated UAT invite email, role `operator`, full name, and future `expiresAt`.
4. Inspect the response shape without printing the returned secret value.

## Expected

Invite/reset/session/apiToken values are not returned after creation or persisted in obvious API responses/logs. The invite creation response should not expose the invite token value in the API response if the Phase 8 expectation is strict.

## Actual

Production returned HTTP 201 and the sanitized response shape included top-level keys `["invite", "token"]`. The actual token value was not printed or stored in UAT evidence.

The token appears to be available only in the immediate admin create response and is not retrievable later via reviewed API routes.

## Evidence

Phase 8 run `20260515202922`:

```json
{
  "scenario": "Secrets are not exposed after creation",
  "httpStatus": { "inviteCreate": 201 },
  "inviteCreateResponseKeys": ["invite", "token"],
  "inviteTokenReturnedAtCreation": true,
  "retrievableLaterViaReviewedApi": false,
  "databasePlaintextTokenColumnFound": false,
  "auditLogTokenValueFound": false,
  "actualSecretValuePrinted": false
}
```

Source/evidence reviewed:

- `backend/src/auth/auth.controller.ts` exposes `POST /auth/invites` as admin-only via `SessionGuard`, `RolesGuard`, and `RequireRoles('admin')`.
- `backend/src/auth/auth.service.ts` returns `{ invite, token }` only from `createInvite(...)`; `acceptInvite(...)` looks up by token hash and does not return the token.
- `backend/prisma/schema.prisma` defines `UserInvite.tokenHash`; no plaintext invite token column is present.
- `backend/src/logs/logs.service.ts` frontend-visible logs select IDs/actions/entities/store/actor summaries, not `beforeData`, `afterData`, or `metadata`.
- `frontend/src/main.tsx` displays the one-time token in the invite creation success panel for the admin; no console/local/session storage logging was found in reviewed source.

## Workaround

Treat invite creation responses as sensitive admin-only material and send the one-time token through a secure channel. Do not log or persist the token value. A code change is required if the product requirement is to avoid returning invite tokens even once in the admin create response.
